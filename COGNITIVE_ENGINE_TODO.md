# Cognitive Engine TODO

## Status

Next-layer planning document.

`Packet` as machine-playable substrate is now strong enough
that external machines should stop focusing only on transport bugs
and start thinking about the next layer above the substrate.

That next layer is currently called:

> cognitive engine

This name is provisional.

## What cognitive engine means here

`Packet` already gives:

- world substrate
- latent / manifest / crystal dynamics
- turbulence as hidden pressure layer
- live runtime behavior

What it does **not** yet give is:

- gameplay pressure
- local meaning of action
- directed choice
- stakes
- higher-order rules over the substrate

That is the job of the cognitive engine.

## Current working hypothesis

The cognitive engine should not begin as:

- more terrain painting
- more visual zones
- more static content

It should begin as:

- operator-driven pressure
- local causality
- decision grammar
- higher-order constraints on a living Packet world

## What external machines should think about

When reading this repository, external machines should now answer:

1. What are the first 3 gameplay-pressure laws that should exist above Packet?
2. Which ProcessLang operators are the best first candidates for that layer?
3. What should be configurable data, and what should stay runtime law?
4. How should local player action become legible cause, not just motion?
5. What is the minimum cognitive layer that turns a living substrate into a game?

## Strong candidate framing
